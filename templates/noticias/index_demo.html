Dúvidas Frequentes do Desenvolvedor

O terminal do PyCharm é como um "Portal Mágico" que te conecta diretamente ao sistema operacional, mas já no diretório do seu projeto.

Passo Didático: Simplesmente procure pela aba **"Terminal"** que fica na parte inferior da sua janela do PyCharm. Clique nela e pronto!

Dica de Mestre: Use o atalho universal Alt + F12 (ou Option + F12 no macOS) para abrir e fechar a aba instantaneamente.

Pense no `.gitignore` como uma "Lista de Exclusão do Detetive Git".

Ele é grande porque precisa listar todos os "arquivos lixo" ou "arquivos temporários" que o Git não deve rastrear:

  • Ambientes Virtuais (venv/): Extremamente grandes e exclusivos do seu computador.
  • Arquivos de Cache (__pycache__, .pytest_cache): Gerados em tempo de execução.
  • Arquivos da IDE (.idea/): Configurações pessoais do seu PyCharm ou VS Code.
  • Segredos (.env): Não devem ser públicos.

O objetivo é garantir que seu repositório Git contenha **apenas o código-fonte essencial**.

O `.env` não subiu porque ele é o "Cofre de Segredos" do seu projeto e, por padrão, está listado no seu `.gitignore`.

Princípio Básico de Segurança:

  • Ele guarda chaves de API, senhas de banco de dados e credenciais que são específicas do ambiente onde o código está rodando.
  • **Nunca** envie dados sensíveis para um repositório público (ou mesmo privado) do Git.

Em vez do `.env`, você deve enviar um arquivo .env.example (ou similar) mostrando quais variáveis são necessárias, mas sem os valores reais.

O manage.py é o "Controle Remoto" do seu Projeto Django.

Ele é um script de linha de comando que:

  • Permite interagir com seu projeto sem precisar configurar variáveis de ambiente complexas.
  • É usado para todas as tarefas administrativas, como: python manage.py runserver (Ligar o servidor) python manage.py makemigrations (Preparar mudanças no banco de dados) python manage.py createsuperuser (Criar um administrador)

Sim, exatamente! A arquitetura do Django segue o princípio de "Projeto como Shopping Center".

  • Um **Projeto** é o grande guarda-chuva: as configurações globais, a URL principal, o arquivo de `settings.py`.
  • Um **Aplicativo (App)** é um módulo específico, uma loja dentro do shopping. Ele faz uma coisa e a faz bem. Ex: `UsersApp`, `BlogApp`, `PaymentsApp`.

Essa modularidade torna seu código mais organizado, reutilizável e fácil de manter.

Com certeza, e interagem muito! Eles são como **"Setores de uma Empresa"** que precisam colaborar para que a empresa funcione.

Exemplo Prático:

  • O `BlogApp` precisa saber quem é o autor de um post. Ele faz isso referenciando o modelo `User` do `UsersApp`.
  • O `PaymentsApp` (o setor financeiro) precisa interagir com o `ProductsApp` (o setor de estoque) para saber o que está vendendo.

A interação se dá através da importação de modelos (from app_name.models import ModelName), chamadas de métodos, e do sistema de Sinais e Mensagens do Django.

O `python-dotenv` é uma "Chave de Leitura de Cofres". Ele é fundamental para a segurança e portabilidade de configurações.

1. Para que Serve? Ele lê as variáveis de configuração (chaves, senhas, URLs) que estão armazenadas de forma segura no seu arquivo `.env` e as injeta no ambiente do seu projeto. Assim, você nunca hardcode informações sensíveis.

2. Como Instalar? Basta executar no seu terminal (com a venv ativa!): pip install python-dotenv

Com isso, você mantém suas credenciais privadas, seguindo as melhores práticas de desenvolvimento.

Uma `venv` é o "Laboratório Isolado" do seu projeto Python.

O Problema: Se você tem o Projeto A que precisa do Django 3.2 e o Projeto B que precisa do Django 4.0, sem uma `venv`, eles entrariam em conflito (o temido "Inferno de Dependências").

A Solução: A `venv` cria um diretório isolado que armazena as cópias exatas do Python e dos pacotes **apenas** para aquele projeto. Assim, o ambiente do Projeto A não interfere no Projeto B.

O processo é simples e utiliza o módulo embutido do Python.

No diretório raiz do seu projeto, execute o **"Comando de Criação do Laboratório"**: python3 -m venv nome_da_venv

Geralmente, o nome padrão dado ao ambiente virtual é apenas venv ou .venv.

Ativar a `venv` é o mesmo que **"Entrar no Laboratório"**. O comando depende do seu sistema operacional:

No Linux ou macOS (Bash/Zsh):

source venv/bin/activate

No Windows (CMD):

venv\Scripts\activate

Você saberá que deu certo quando o nome da sua `venv` ((venv)) aparecer no início da linha de comando do seu terminal.

O `requirements.txt` é a "Receita Secreta" do seu projeto, listando todos os ingredientes (pacotes) e suas versões exatas.

Com sua `venv` ativa, execute o **"Comando da Receita"** no terminal: pip freeze > requirements.txt

Este comando congela (`freeze`) a lista de pacotes instalados e redireciona (`>`) essa saída para o arquivo `requirements.txt` na raiz do projeto.

Seu colega de trabalho acabou de te enviar o código. Para "montar" o projeto na sua máquina, siga os passos (após criar e ativar sua `venv`!):

Execute o **"Comando de Montagem"** na pasta raiz do projeto: pip install -r requirements.txt

O flag `-r` instrui o `pip` a ler e instalar todos os requisitos listados no arquivo. Isso garante que você tenha as mesmas dependências que o projeto original.
